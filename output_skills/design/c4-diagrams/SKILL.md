# C4 Diagrams

C4 models software architecture as a hierarchy of zoom levels. Each level answers a different question for a different audience. Most teams need only the top two levels.

## Abstractions

Four abstractions, from coarsest to finest:

**Person** — a user, actor, or role that interacts with the system.

**Software System** — the highest level. A system that delivers value to its users. Your system is one; the things it depends on are others.

**Container** — an independently deployable/runnable thing within a system: a web app, API, database, message queue, file store, serverless function. NOT a Docker container. The name is unfortunate — think "deployment unit" or "process."

**Component** — a grouping of related code within a container, accessible through a defined interface. A module, package, or service layer.

## Choosing the Right Level

```
"Who uses this system and what else does it talk to?"
  → System Context diagram

"What are the major moving parts inside?"
  → Container diagram

"How is this container organized internally?"
  → Component diagram (only if it adds value — often not worth maintaining)

"What does the code look like?"
  → Code diagram (almost never — IDEs do this better)

"How do these parts interact at runtime for feature X?"
  → Dynamic diagram (use sparingly — only for non-obvious flows)

"Where does this run in production?"
  → Deployment diagram
```

System Context and Container are recommended for every system. Component only when the internal structure is non-obvious and the diagram will be maintained. Code diagrams should be auto-generated by tools, not hand-crafted.

## Workflow

### From Existing Code

1. Explore the codebase to identify the system boundary, external dependencies, and internal structure
2. Start with a System Context diagram — identify people and external systems
3. Zoom into a Container diagram — identify the deployable units and data stores
4. Ask the user if deeper levels are needed before going to Component
5. Ask which output format they prefer (Structurizr DSL, Mermaid, or ASCII) — default to Structurizr if they have no preference

### From Designs Being Planned

1. Interview the user: what is the system? Who uses it? What does it integrate with?
2. Draft a System Context diagram and validate with the user
3. Discuss the major containers — what tech, what responsibilities
4. Draft a Container diagram
5. Go deeper only if the user wants to explore a specific container's internals

### General Principles

- Always start at System Context and zoom in. Never start at Component level.
- Each diagram should tell a story to its audience. System Context is for everyone; Container is for technical people; Component is for developers on that container.
- Keep diagrams focused. A Container diagram that shows 20+ containers is a warning sign — consider grouping or splitting.
- Show only the relationships that matter for the story. Not every possible connection.
- Label relationships with what flows or what happens ("sends orders to", "reads from"), not just "uses."
- Include technology choices on Container and Component diagrams (e.g., "Spring Boot API", "PostgreSQL 15").

## Output Format References

Syntax and conventions for each rendering format:

- [Structurizr DSL](references/formats/structurizr.md) — full modeling language, renders at playground.structurizr.com
- [Mermaid](references/formats/mermaid.md) — embeds in markdown, renders in GitHub/docs
- [ASCII](references/formats/ascii.md) — inline in code, commits, or terminal output

Read the relevant reference before generating output in that format.

When generating Structurizr DSL, produce a complete `workspace.dsl` file that the user can paste into playground.structurizr.com to see the rendered diagram immediately.

When generating ASCII diagrams, pipe the output through `uv run scripts/check_ascii_alignment.py` to verify box alignment before presenting. Misaligned right walls are the most common mistake in ASCII diagrams.

## Anti-Patterns

- Labeling a diagram "Container" but showing software systems, not containers — mismatch between level and content
- Showing deployment details (load balancers, clusters, replicas) on a Container diagram — that belongs in a Deployment diagram
- Using implementation names ("OrderServiceImpl", "handleRequest") instead of descriptive names ("Order Service", "processes customer orders")
- Creating Component diagrams for every container — only diagram the ones where internal structure isn't obvious
- Including every relationship — cluttered diagrams communicate nothing. Show what matters for the story.
- Forgetting to show the external context — containers don't exist in a vacuum. Show the people and external systems that connect to them.
